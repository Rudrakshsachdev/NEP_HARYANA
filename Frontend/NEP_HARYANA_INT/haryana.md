1. INTRODUCTION
In the current system, colleges are required to submit their academic and institutional data
to the Haryana Government in the form of documents, PDFs, and reports for evaluation
under different schemes and policies such as NEP 2020. This process is mostly manual,
time-consuming, and difficult to manage. The data is often scattered across multiple files,
which makes it hard for authorities to review, compare, and analyze information
efficiently.

The government also follows a structured evaluation system where colleges are assessed
based on various criteria such as academic performance, faculty strength, research
activities, infrastructure, and student outcomes. Each parameter requires proper data
along with supporting documents as proof. However, due to the lack of a centralized
system, the entire process becomes slow and prone to errors.

To solve these problems, this project proposes the development of a web-based academic
evaluation system. The system will allow colleges to submit their data through structured
online forms instead of manual documents. It will also enable automatic calculation of
scores based on predefined criteria, along with document upload for verification.

The platform will provide dashboards and reports to help government authorities easily
analyze data, compare institutions, and make informed decisions. By digitizing the entire
process, the system aims to improve efficiency, accuracy, transparency, and ease of use
for both colleges and administrators.

Overall, this project focuses on transforming a traditional, document-based evaluation
system into a modern, organized, and automated digital solution.

2. SCOPE
The scope of this project focuses on developing a web-based system that simplifies and
digitizes the process of academic data submission and evaluation for colleges under
schemes like Model Sanskriti and the National Education Policy (NEP) 2020
Implementation Excellence Award.

The system will allow colleges to enter their institutional data through structured online
forms instead of submitting physical documents or PDFs. It will support different types
of data such as basic college information, academic details, faculty data, student statistics,
and NEP-related indicators like internships, research activities, innovation cells, and
policy implementation. As seen in the official format, each indicator also requires

Platinum, Gold, and Silver. This will help authorities quickly identify high-performing

5. FUNCTIONAL REQUIREMENTS
The functional requirements define the core functionalities that the system must perform
to meet its objectives. These requirements describe how users will interact with the
system and how the system will respond to different actions.

FR1: User Registration
The system shall allow new users (colleges) to register by providing required details such
as college name (selected from a predefined list), email, phone number, password, and
address. The system shall validate the input data and ensure that the email is unique
before creating a new account.

FR2: User Login
The system shall allow registered users to log in using their email and password. Upon
successful authentication, users shall be redirected to their respective dashboard based on
their role.

FR3: Role-Based Access Control
The system shall support different user roles such as college users and administrators.
Each role shall have specific permissions. Colleges can submit and manage their data,
while administrators can review, evaluate, and manage all submissions.

FR4: Data Entry and Submission
The system shall allow college users to enter and submit data through structured forms.
The data shall include institutional details, academic information, faculty data, student
data, and NEP-related indicators.

FR5: Indicator-Based Input Handling
The system shall collect data based on predefined indicators provided by the government.
It shall support different types of inputs such as Yes/No options, numeric values, and text
fields.

FR6: Document Upload
The system shall allow users to upload supporting documents for each data entry. These
documents shall be linked to the respective indicators for verification purposes.

FR7: Data Validation
The system shall validate all input data to ensure correctness. It shall enforce required
fields, proper formats, and logical constraints before allowing submission.

FR9: Automated Evaluation and Scoring

FR10: Result Classification

FR11: Dashboard Display

8. SYSTEM ARCHITECTURE
The proposed system follows a three-tier architecture consisting of the frontend, backend,
and database layers. This architecture helps in organizing the system efficiently and
ensures scalability, maintainability, and smooth data flow.

Frontend Layer (User Interface)
The frontend is the part of the system that users interact with. It is developed using
modern web technologies such as React.js or Next.js. This layer is responsible for:

 User registration and login
 Displaying forms for data entry
 Showing dashboards, charts, and reports
 Handling user interactions
HSHEC
College users and administrators access the system through a web browser, and the
interface is designed to be simple and user-friendly.

Backend Layer (Application Logic)
The backend is the core of the system, developed using Django and Django REST
Framework. It handles:

 User authentication and authorization
 Processing of submitted data
 Implementation of business logic
 Automated evaluation and scoring
 Managing role-based access
The backend receives requests from the frontend, processes them, and sends responses
back in a structured format (usually JSON).

Database Layer (Data Storage)
The database layer is responsible for storing all system data. It is implemented using
PostgreSQL. This layer stores:

 User information
 College data
 Academic and institutional records
 Uploaded documents
 Evaluation results
The database ensures that data is stored securely, consistently, and can be retrieved
efficiently.

Communication Between Layers
The frontend and backend communicate through RESTful APIs. When a user performs
an action (such as submitting a form), the frontend sends a request to the backend. The
backend processes the request, interacts with the database if needed, and returns the result
to the frontend.

CLIENT REQUIREMENTS FROM HARYANA GOVERNMENT
The Haryana Government acts as the primary client for this system and defines the
overall structure, evaluation criteria, and data requirements. The system is designed based
on the guidelines, formats, and policies provided by the government. The following are
the key requirements from the client:

Evaluation Criteria and Indicators
The government must provide predefined evaluation parameters and indicators based on
which colleges will be assessed. These include NEP-related activities such as internships,
research, innovation, academic reforms, and institutional development. Each indicator is
associated with specific marks and scoring rules, which are essential for implementing
the automated evaluation system.

Standardized Data Format (Proforma)
The government provides a fixed format (proforma) that defines what data needs to be
collected. This includes structured fields such as institutional details, indicator-based
inputs, and required documents. The system must strictly follow this format to ensure
uniform data collection across all colleges.

Master List of Colleges
The government must provide an official list of colleges in Haryana along with details
such as college name, AISHE code, type, and location. This ensures that only valid
institutions can register and submit data in the system.

Data Validation Rules
The client must define rules for validating data, such as required fields, acceptable value
ranges, and formats. These rules help maintain accuracy, consistency, and reliability of
the submitted data.

Document Requirements
The government specifies the types of supporting documents required for each indicator,
such as certificates, MOUs, policy documents, and activity proofs. The system must
allow uploading and linking of these documents with the corresponding data entries.

Scoring and Classification Rules
The government defines how scores are calculated and how institutions are categorized
(e.g., Platinum, Gold, Silver). These rules are necessary for implementing automated
result generation.

Workflow and Submission Process
The client defines the process flow, including data submission, review, and final
evaluation. The system must support status tracking such as submitted, under review,

Reporting and Dashboard Requirements
The government specifies the type of reports and dashboards required for analysis. This
includes summary reports, performance comparison, and data visualization to support
decision-making.

Security and Compliance Guidelines
The system must follow security standards defined by the government, including secure
login, data protection, and restricted access based on user roles.

Timeline and Submission Deadlines
The government provides deadlines for data submission and evaluation. The system must

period.
